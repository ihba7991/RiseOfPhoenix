The 8 to know cold

Singleton · Factory Method · Builder · Adapter · Decorator · Strategy · Observer · State. If asked "design X," these cover ~80% of LLD answers. Strategy + State + Observer alone solve a huge fraction of "design a vending machine / parking lot / order system" prompts.

Fast disambiguation — the pairs people confuse

  • Strategy vs State: Strategy = you pick the algorithm and it doesn't change itself. State = the object transitions itself between behaviors.
  • Decorator vs Proxy: Both wrap. Decorator adds behavior; Proxy controls access (same behavior, gated).
  • Factory Method vs Abstract Factory: Factory Method = one product, subclass decides. Abstract Factory = a family of products that must be consistent.
  • Adapter vs Facade: Adapter makes one incompatible thing fit; Facade simplifies many things behind one interface.
  • Decorator vs Chain of Responsibility: Decorator — everyone in the stack acts. Chain — one handler acts, then it stops (or passes).

Read the arrows: <|.. implements · <|-- inherits · o--> "has a" reference · ..> uses/creates.

Worked example — the GMan project (this repo)

A robot (Gman) moves Source → Destination; we compute remaining power. Clean Entity / Model / Utils layering — a good "show me how you'd structure a small LLD" answer.
classDiagram
    class Gman {
        -Source source
        -Destination destination
        -String directionFacing
    }
    class Source {
        -int coordinateX
        -int coordinateY
    }
    class Destination {
        -int coordinateX
        -int coordinateY
    }
    class CostModel {
        -int travellingCost
        -int turningCost
        -int totalAvailablePower
    }
    class PowerConsumedCalculator {
        -Gman gman
        -CostModel costModel
        +calculatePowerConsumed() int
        +remainingPower() int
    }
    class Main {
        +main(args)
    }
    Gman o--> Source
    Gman o--> Destination
    PowerConsumedCalculator o--> Gman
    PowerConsumedCalculator o--> CostModel
    Main ..> Gman
    Main ..> CostModel
    Main ..> PowerConsumedCalculator
Why it scores: entities hold state, the calculator holds logic (SRP) · clear package layering · a new command is a new case, a different cost scheme is a new CostModel — a natural seam to introduce Strategy if turn/distance rules diverge.

Concurrency — Java backend cheat sheet

Core mental model
  • Race condition: outcome depends on thread timing on shared mutable state.
  • Visibility vs atomicity: volatile gives visibility but NOT atomicity (i++ is still 3 ops). synchronized/locks give both.
  • Happens-before: JMM rule that makes one thread's writes visible to another — via synchronized, volatile, Thread.start/join, final fields.

Deadlock — break any one of the 4
Mutual exclusion · Hold-and-wait · No preemption · Circular wait.  Most common fix: global lock ordering so circular wait can't form, or tryLock + timeout + back off.
Patterns interviewers ask you to build
  • Producer–Consumer → BlockingQueue (clean) or wait/notify in a while loop.
  • Bounded resource / connection pool → Semaphore(n).
  • Rate limiter → token bucket with AtomicLong + timestamps, or Semaphore + scheduled refill.
  • Thread-safe Singleton → enum (simplest) or double-checked locking with volatile instance.
  • Wait for parallel tasks → CountDownLatch (one-shot) or CompletableFuture.allOf.
Double-checked locking Singleton — know how to write this:
class Config {
  private static volatile Config instance;   // volatile is REQUIRED
  private Config() {}
  static Config get() {
    if (instance == null) {                  // 1st check, no lock
      synchronized (Config.class) {
        if (instance == null) {              // 2nd check, under lock
          instance = new Config();
        }
      }
    }
    return instance;
  }
}
Why volatile: without it another thread can see a non-null but partially-constructed object due to instruction reordering.
Gotchas to say out loud in interviews
  • i++ is not atomic. volatile won't save it — use AtomicInteger or a lock.
  • HashMap under concurrent writes can corrupt / infinite-loop — use ConcurrentHashMap.
  • Don't swallow InterruptedException — restore the flag: Thread.currentThread().interrupt().
  • Prefer high-level (ExecutorService, BlockingQueue, CompletableFuture) over raw threads + wait/notify. Signals senior judgment.
